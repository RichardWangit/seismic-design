所有回覆必須使用繁體中文。無論使用者以何種語言提問,Claude 的所有回應一律以繁體中文書寫,包含說明、程式碼注解建議及錯誤訊息解釋。

介面配色採用「典籍書卷」設計。新增 UI 元件時應遵循此配色系統,避免引入與古典書卷美學不符的現代高飽和色彩。

# 工作準則

⚠️ 注意:以下這些準則偏向謹慎而非速度。對於瑣碎的任務,請自行判斷。

## 1. 先思考再撰碼

不要假設。不要隱藏困惑。揭示取捨。

實作之前:
- 明確陳述你的假設。若有不確定,請先詢問。
- 若存在多種解讀,逐一列出——不要私自選擇。
- 若有更簡單的方法,請說出來。必要時應予以反駁。
- 若有不清楚之處,停下來。說明困惑所在。提出問題。

## 2. 簡潔優先

最少的程式碼解決問題。不做臆測性的預先實作。

- 不實作超出需求的功能。
- 不為單次使用的程式碼建立抽象層。
- 不實作未被要求的「彈性」或「可設定性」。
- 不為不可能發生的情境撰寫錯誤處理。
- 若你寫了 200 行而 50 行即可解決,請重寫。

自問:「資深工程師會覺得這過於複雜嗎?」若是,請簡化。

## 3. 精準修改

只動必要的部分。只清理你自己製造的混亂。

編輯現有程式碼時:
- 不要「改善」相鄰的程式碼、注解或格式。
- 不要重構沒有問題的部分。
- 遵循現有的程式風格,即使你有不同偏好。
- 若發現不相關的廢棄程式碼,提及即可——不要擅自刪除。

若你的修改產生了孤立元素:
- 移除因你的修改而變得無用的 import / 變數 / 函式。
- 不要刪除修改前既有的廢棄程式碼,除非被要求。

檢驗標準:每一行修改都應能直接追溯至使用者的需求。

## 4. 目標導向執行

定義成功標準。持續循環直到驗證通過。

將任務轉化為可驗證的目標:
- 「新增驗證」→「為無效輸入撰寫測試,再讓測試通過」
- 「修復 bug」→「撰寫能重現問題的測試,再讓測試通過」
- 「重構 X」→「確保重構前後測試均通過」

對於多步驟任務,請說明簡要計畫:

1. [步驟] → 驗證:[確認項目]
2. [步驟] → 驗證:[確認項目]
3. [步驟] → 驗證:[確認項目]

明確的成功標準讓你能獨立循環作業。模糊的標準(「讓它跑起來」)則需要不斷釐清。

## ✅ 這些準則奏效的跡象

- diff 中不必要的修改更少
- 因過度複雜而需要重寫的情況減少
- 釐清性問題在實作之前提出,而非在出錯之後
